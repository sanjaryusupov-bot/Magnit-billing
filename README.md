# MCOS — дашборд «Статус по отгрузкам с РЦ» (Streamlit)

## Что внутри

3 вкладки + экспорт:

1. **📅 Отгрузки по дням** — День / Кол-во шт / Итого сумма (график + таблица) с переходом
   в детали по конкретному дню: Дата, Магазин, Кол-во шт, Сумма, Сумма за шт.
2. **🗺️ Сводная (Город / Транспорт / Рейс)** — Город, Транспорт, Рейс, Итого сумма, Кол-во шт,
   Сумма за шт. Город берётся из листа «Статус по заказам» (по номеру заказа), с фолбэком
   на колонку «Регион» из «Статус по отгрузкам», если заказ не нашёлся.
3. **⏱️ SLA: план vs факт** (доп. функционал) — сравнивает плановую дату отгрузки
   (лист «Статус по заказам») с фактической (лист «Статус по отгрузкам») по заказам `1M-...`.
   Если колонка «Дата отгрузки план» пустая — берётся дата раздела
   (`01/07/2026 (заказы на ср)`), т.к. в вашем файле план часто указан только там.
4. **📤 Экспорт в Excel** — полный отчёт одним файлом, включая лист-реестр для аудита
   (см. ниже).

Глобальные фильтры в сайдбаре: период и фильтр «только заказы 1M-...».

## Разбивка маршрутов внутри «Рейс N»

Новый маршрут (`Маршрут_ID`) начинается, когда в строке ненулевая «1 точка»,
ИЛИ когда регион (тарифная зона) этой строки отличается от региона текущего
маршрута — например, чередование Наманган/Фергана внутри одного «Рейс N» не
может быть одним физическим маршрутом, даже если «1 точка» не проставлена.
Проверено на реальных данных с чередующимися регионами — каждый заказ в
таком случае корректно получает собственный `Маршрут_ID`.

## Про живые сводные таблицы Excel (двойной клик → детали)

Полностью автоматически сгенерировать «живую» сводную таблицу Excel (с
родным двойным кликом «Показать детали») из кода надёжно не получилось —
это требует специфичного XML (pivotCache/pivotTable), который открытые
Python-библиотеки (openpyxl) не поддерживают для создания с нуля, а
инструмент для гарантированно корректной генерации (LibreOffice headless)
в этой рабочей среде оказался нестабилен. Рисковать битым файлом не стали.

Вместо этого лист «Реестр» сохранён как настоящая Excel-таблица
(`ТаблицаРеестр`) — благодаря этому построить сводную таблицу вручную
занимает 30 секунд (Вставка → Сводная таблица → OK, источник подставится
сам), и такая сводная уже полностью поддерживает двойной клик. Точные поля
для каждого листа — во вкладке «📤 Экспорт в Excel» приложения, в блоке
«Как за 2 клика построить живую сводную таблицу».

## Экспорт в Excel — формат «Биллинг»

Вкладка «📤 Экспорт в Excel» — основная кнопка формирует файл с одним листом
**Реестр** — построчно по каждому заказу: Дата отгрузки, Заказ, Рейс
(Маршрут_ID), Кол-во штук, ID магазина, Магазин, Адрес, Координаты, Регион,
Регион тариф, Авто, 1 точка, 2 точка и далее, Грузчик экспедитор, Итого
сумма. Оформлен как настоящая Excel-таблица (`ТаблицаРеестр`) — на её основе
за пару кликов строится сводная таблица (см. инструкцию в самом
приложении).

**Регион тариф** — это регион, приведённый к тарифной зоне (например,
Янгиюль/Ангрен/Ташобласть-Чирчик → Ташкент, Коканд → Фергана). Маппинг
взят из примера заказчика и зашит в `DEFAULT_REGION_TARIFF_MAP` (в
`data_parsing.py`) — если появятся новые спутниковые города, добавьте их
туда.

Ниже на той же вкладке — второй, расширенный вариант отчёта (с суммой,
честно распределённой по кол-ву штук на заказ, и отдельным листом-аудитом),
см. следующий раздел.

## Как считается стоимость доставки (важно)

В исходнике колонка «Итого сумма» — это тариф за точку маршрута (магазин),
проставленный **один раз на уникальный магазин в рейсе**: повторный заказ в
тот же магазин в тот же рейс получает 0 (доставка уже учтена). Из-за этого
при простом суммировании «по магазину» суммы получаются неровными.

Правильный расчёт (реализован в `add_route_economics()` в `data_parsing.py`):

1. **Сумма маршрута** = сумма «Итого сумма» по всем строкам рейса (сумма
   тарифов по каждой уникальной точке).
2. **Кол-во шт маршрута** = сумма «Кол-во шт» по всем строкам рейса, включая
   повторные заказы в тот же магазин.
3. **Тариф за шт** = Сумма маршрута / Кол-во шт маршрута.
4. Для каждого заказа: **Сумма распределённая** = Кол-во шт этого заказа ×
   Тариф за шт — стоимость доставки размазывается по всем штукам рейса
   пропорционально, а не концентрируется на первой точке.

Пример из вашего рейса №3: пять уникальных точек (Риёзи, Дархан Полянка,
Сокольная, Коленная, Аппаз) с тарифами 450 000 + 165 000×4 = 1 110 000.
При общем кол-ве штук в рейсе 7 523 тариф за штуку = 1 110 000 / 7 523 ≈
147,54. Дальше сумма каждого магазина = его кол-во штук × 147,54.

Итог по дню/рейсу в сумме не меняется (это те же деньги, просто честно
распределённые) — меняется только разбивка по магазинам/заказам.

## Экспорт в Excel и лист-реестр

Вкладка «📤 Экспорт в Excel» формирует файл с листами:

- **Отгрузки по дням** — как в первой вкладке, за весь период.
- **Детали по магазинам** — Дата, Магазин, Кол-во шт, Сумма, Сумма за шт —
  за весь период (не только выбранный день).
- **Сводная** — Город / Транспорт / Рейс / Итого сумма / Кол-во шт / Сумма за шт.
- **SLA план-факт**.
- **Реестр (аудит)** — построчно по каждому заказу в каждом рейсе: исходный
  тариф из таблицы, сумма и кол-во штук по всему рейсу, тариф за штуку и
  итоговая распределённая сумма. По этому листу можно вручную пересчитать
  и проверить любую цифру из отчёта.
- **Как считается** — короткое текстовое описание формулы.

## Доступ к данным — через сервисный аккаунт Google (не публичная ссылка)

Таблица остаётся закрытой для посторонних. Доступ на чтение получает только
сервисный аккаунт, от имени которого работает приложение.

### Шаг 1. Создайте / получите сервисный аккаунт

Google Cloud Console → IAM & Admin → Service Accounts → Create Service Account,
затем на вкладке **Keys** → Add Key → Create new key → **JSON**. Скачается файл
с полями `project_id`, `private_key`, `client_email` и т.д.

⚠️ Этот JSON-файл — секрет. Никогда не публикуйте его (в чат, в открытый
репозиторий и т.п.). Если ключ где-то засветился — сразу удалите его в Google
Cloud Console (Keys → тот ключ → Delete) и создайте новый.

### Шаг 2. Включите Google Sheets API

В том же проекте GCP: APIs & Services → Library → найдите **Google Sheets API**
→ Enable. (Google Drive API включать не обязательно, но не помешает.)

### Шаг 3. Дайте доступ сервисному аккаунту к таблице

Откройте вашу Google-таблицу → «Настройки доступа» → «Добавить пользователей» →
вставьте email сервисного аккаунта (поле `client_email` из JSON, вида
`xxx@your-project.iam.gserviceaccount.com`) → роль **Читатель**. Таблицу *не*
нужно открывать по публичной ссылке.

### Шаг 4. Положите ключ в Secrets приложения

Локально: скопируйте `secrets.toml.example` в `.streamlit/secrets.toml` и
впишите туда реальные значения из JSON-файла (секцию `[gcp_service_account]`).
Добавьте `.streamlit/secrets.toml` в `.gitignore` — коммитить его нельзя.

На Streamlit Community Cloud: **Manage app → Settings → Secrets** — вставьте
туда то же содержимое (в формате TOML, как в `secrets.toml.example`) и
сохраните. Приложение перезапустится само.

## Как запустить локально

```bash
pip install -r requirements.txt
streamlit run app.py
```

Откроется на `http://localhost:8501`. В сайдбаре укажите ID таблицы (кусок
ссылки между `/d/` и `/edit`, по умолчанию уже подставлен ваш) и нажмите
«Загрузить / обновить данные».

## Если что-то не грузится

- **"Не настроены Secrets"** — не добавлен `[gcp_service_account]` в Secrets
  (см. Шаг 4).
- **"Нет доступа к таблице"** — сервисному аккаунту не выдали доступ на
  чтение (см. Шаг 3), либо не включён Google Sheets API (см. Шаг 2).
- **"Не найден один из листов"** — в таблице переименованы вкладки
  `Статус по отгрузкам` / `Статус по заказам`.

## Если структура листов немного другая

Вся логика парсинга — в `data_parsing.py`, в функциях `parse_shipments()` и
`parse_orders_status()`. Если поменяется порядок/название колонок в
исходнике, достаточно поправить там номера колонок — остальной код (UI,
агрегации, доступ к API) трогать не нужно.

## Деплой на Streamlit Cloud — если раньше падала установка зависимостей

В `requirements.txt` версии жёстко зафиксированы (`==`), а в `runtime.txt`
указан Python 3.12 — это устраняет типичную ошибку "Error installing
requirements", возникающую на слишком новых версиях Python без готовых
бинарных пакетов (wheels).


## Если структура листов немного другая

Вся логика парсинга — в `data_parsing.py`. Если поменяется порядок/название
колонок в исходнике, достаточно поправить номера колонок в
`parse_shipments()` и `parse_orders_status()` — остальной код (UI, агрегации)
трогать не нужно.
